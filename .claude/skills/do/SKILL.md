# /do — End-to-End Orchestrator (Raqiy — Онлайн-школа рукии)

Ты ведёшь полный цикл фичи/контента, в духе multi-agent app builders (Emergent-style
architect → developer → QA → review), но под особенности именно этого проекта:
**статичный сайт** (GitHub Pages: HTML/CSS/JS в `pages/`/`design/`, без Electron, без
своих Node-агентов/ботов — интеграция с внешним бэкендом через `integration/`, см.
project.md §11/§16) **плюс курс религиозного контента**, где качество и безопасность
текста напрямую влияют на реальных пациентов, которых будущие выпускники будут лечить
через RUKYA Pro (project.md §7).

Смысл скилла тот же, что у прототипа: **ничего не доходит до пользователя как
"готово", пока не проверено end-to-end.** Пропуск стадии — это ровно тот сбой, для
предотвращения которого скилл существует. Если стадия реально не применима — скажи
об этом явно, не пропускай молча.

## Стиль работы (ВАЖНО — перекрывает многословность)

**Работай молча, без воды. Не объясняй по ходу — просто делай.**
- Никаких вступлений, планов вслух, «сейчас я сделаю…», перечисления опций,
  которые не будешь делать. Не пересказывай прочитанный контент/код.
- Не анонсируй каждую стадию абзацем. Стадии проходишь молча (одна короткая
  строка-заголовок на стадию максимум, и только если это реально помогает следить).
- Вопрос задавай ТОЛЬКО если без ответа нельзя двигаться — один раз, коротко.
- Весь «рассказ» — в финальном итоге, а не по дороге.

## Финальный итог (ОБЯЗАТЕЛЬНО — единственная развёрнутая часть)

```
✅ Готово: <одна строка что сделано>

Было:  <как было — 1-2 строки>
Стало: <как стало — 1-2 строки>

Изменено: <file:line, …>
Проверено: <чем — /verify, webapp-testing, сверка цитат по §7/§9>
```

- Если что-то не доделано/красное — назови явно строкой «⚠️ Осталось: …».
- Если правка касается контента модулей — добавь строку «📖 Статус: черновик, ждёт
  подтверждения шейха» либо «📖 Статус: подтверждено пользователем как шейхом» —
  в зависимости от того, было ли явное подтверждение в этом же диалоге.
- **Не создавай отдельные `.md`-отчёты сверх запрошенного.** Итог — в чате. Правки в
  сам `project.md` — не «отчёт», а часть содержимого проекта (project.md — рабочий
  журнал решений этого проекта, см. ниже), это не подпадает под запрет.

## Hard rules этого проекта (перекрывают любой суб-скилл)

- **Разрешение на код проверяется, не предполагается.** Стандартный режим этого
  проекта — контент/план фиксируются в `project.md`, а вёрстка/код не пишутся, пока
  пользователь явно не скажет «пиши код»/«верстай»/аналог (см. историю разговора —
  это правило действовало всю сессию). Если из текущего диалога не следует явного
  разрешения на конкретный кусок кода — спроси один раз на стадии Architect, прежде
  чем переходить к Build. Правки самого `project.md` (план, черновики контента,
  статусы триажа) это разрешение не требуют — это ровно то, что уже делалось всю
  сессию.
- **Контент модулей курса (`content/module-*/`) подчиняется критерию §7/§9
  project.md:** каждый фрагмент — либо (а) прямая цитата аята/хадиса с указанием
  источника, либо (б) устоявшаяся практика (аят аль-Курси, три защитные суры,
  общепризнанные азкары), либо (в) авторская методика — маркируется явно
  («авторский метод, не сертифицируется как Сунна») и не идёт в сертифицируемые
  модули без явного решения пользователя. Никогда не подавай (в) как (а)/(б).
- **Красные линии диагностики и работы с пациентом (§9а project.md) — абсолютны,
  не подлежат смягчению «по стилю автора»:** диагностика никогда не строится на
  пальпации/надавливании на тело пациента (особенно репродуктивные органы); ни один
  протокол не инструктирует игнорировать выраженное пациентом страдание/мольбу как
  «манипуляцию духа» или физически удерживать пациента иначе чем для предотвращения
  самотравмирования. Это применяется к ЛЮБОМУ новому контенту, не только к уже
  протриаженным 23 книгам.
- **Дедупликация обязательна (§9б project.md):** прежде чем писать новую страницу
  книги/модуля, проверь, не покрыта ли уже та же таблица/формула/хадис в общем
  «Справочнике» — не плоди повторы.
- **Мобильный экран — основной** (§13): вёрстка сначала под узкий вьюпорт (реальный
  вьюпорт в webapp-testing, не просто сужение окна десктопа), десктоп — расширение.
  Масштабирование шрифта — через CSS-переменную по паттерну `--rp-fs` из RUKYA Pro
  (`rem`, не `px`) — уже согласовано переиспользовать (§8.2), не изобретать заново.
- **Один источник дизайн-токенов.** Палитра золото/пергамент (учебный контент) +
  зелёный (кабинет ученика) / графит (кабинет админа) уже утверждена (§16а). Все
  цвета/скругления/шрифты — через `design/tokens.json`→`tokens.css`
  (design-system skill), не инлайновые стили, дублирующие вторую палитру.
- **`prefers-reduced-motion` уважается** во всех анимациях фона/переходов (§21).
- **Оплата и способ регистрации пока НЕ решены** (§16/§18 project.md, открытые
  пункты). Если фича их касается — Architect-стадия останавливается и спрашивает,
  не выбирает по умолчанию.
- Раздел `content/archive/` (книги вне сертификации) — публикуется с явной пометкой
  статуса, не тихо смешивается с сертифицированными модулями (§8.1).

**Дизайн-полировка (сверка с эталоном).** Источник истины по виду экранов —
`Онлайн-школа рукии - концепция.html` (принятый макет, §16а) плюс решения по нему
(1b дорожка-путь, 2a тёмная карточка аята, §16а). Запускай, когда просят «приведи к

1. Сопоставь макет ↔ экран сайта по названию блока (главная/оглавление модулей,

2. Сравни РЕНДЕРОМ (webapp-testing screenshot), не по коду — эталонный `file://`

3. Отличай дизайн-язык (цвет/форма/шрифт/раскладка) от плотности контента — если
   живой экран содержит больше книг/полей, чем упрощённый мок, это не расхождение,
   не удаляй функционал ради сходства.
4. Правь узко — `design/`/CSS-токены + точечные Edit в `pages/`, тексты — в
   `content/`, не хардкодь в HTML.
5. Подтверди глазами повторным скриншотом после правки.

**Маршрут B — Оптимизация:** найти реальное узкое место (прочитать/измерить, не
гадать) → /simplify → /code-review high → /security-review (если тронута

**Маршрут C — Новый модуль/книга курса (контентная работа, не код):**
1. **Проверь исходник по критерию §7/§9** — что прямая цитата Корана/Сунны, что
   устоявшаяся практика, что авторская методика. Зафиксируй в `content/triage.md`
   и в таблице §8 project.md.
2. **Проверь дедупликацию (§9б)** — совпадает ли что-то с уже вынесенным в
   «Справочник» (словарь органов, таблицы классификации, базовые азкары) — если да,
   не пересказывай заново, ссылайся.
3. **Проверь красные линии** — нет ли пальпаторной диагностики, нет ли инструкций
   игнорировать страдание пациента. Если есть — исключи или перепиши по примеру

4. **Напиши текст** в `content/module-N/` — с явной пометкой источника каждого
   раздела (аят/хадис/устоявшаяся практика/авторский метод-архив).
5. **Пометь статус.** Новый теологический текст всегда помечается «черновик, ждёт
   проверки» в самом файле, пока пользователь прямо не подтвердит его как
   финальный — это не бюрократия, это единственный гейт, который в этом проекте
   сознательно не автоматизирован (project.md §6, §12).
6. Обнови сводные таблицы в project.md (§4, §8, §9а/б/в) — это и есть «запись
   решения» для этого проекта (см. ниже, у Raqiy нет отдельного adr/project-memory
   скилла — `project.md` сам служит этим журналом).
7. Если контент требует новой страницы — маршрут A для вёрстки.

проекте — просто новый пункт/раздел там же).

**Маршрут E — Фича портала (dashboard, квиз, регистрация, сообщения и т.п.):**
пайплайн стадий 1–6 ниже.

**Маршрут F — Fix:**
1. Собери список проблем (из QA-прогона, code-review, аудита, жалобы).
2. Отсортируй по severity: CRITICAL (небезопасный/сфабрикованный контент,

## Параллельные фоновые агенты

У этого проекта пока нет выделенных именных субагентов (в отличие от RUKYA Pro —
там `.claude/agents/rukya-spa-dev` и т.п.). Если задача дробится на независимые
куски (например: вёрстка одной страницы + независимая проверка цитат в другой
книге), запускай через Agent tool несколько `general-purpose`/`Explore`-агентов
**одним сообщением** (несколько вызовов Agent = параллельно), каждому — только его
под-объём. Если такой параллельный паттерн станет частым — есть смысл завести
проектные субагенты по образцу RUKYA Pro (например `raqiy-site-dev` для
`pages/`/`design/`, `raqiy-content-dev` для `content/` с автоматическим применением
критерия §7/§9) — но заводить их сейчас не буду, пока не спросили отдельно.

## The pipeline (маршруты C и E)

### 1. Architect (scope + plan + разрешение на код)
- Сформулируй фичу в одну строку. Неоднозначность — спроси ОДИН раз, потом веди.
- Определи, какой слой затронут: сайт (`pages/`, `design/`, `content/`), интеграция
  (`integration/` — вызовы внешнего API авторизации/данных, ещё не выбран
  окончательно, §16), или чисто контент курса (маршрут C, без кода).
- **Если это код (не только project.md/content-черновик) — проверь, было ли явное
  разрешение писать код в этом диалоге.** Если нет — спроси один раз перед Build.
- Для нетривиальной фичи — **Plan**-агент/**EnterPlanMode** для конкретных шагов и
  ключевых файлов. Для мелкой правки — 2–3 строки инлайн-плана достаточно.

### 2. Design check (только для нового/меняющегося UI)
Если фича добавляет/меняет пользовательский экран — свериться с макетом (см.
«Дизайн-полировка» выше) и принятыми решениями §16а перед кодом. Для чисто
контентных/бэкенд-интеграционных правок — пропустить.

### 3. Build
- Мелкая однослойная правка — делай сам инлайн. Крупная/многослойная — раздели на
  независимые куски и раздай параллельно (см. «Параллельные фоновые агенты»).
- **Edit** поверх существующих файлов, где возможно. Текст — в `content/`, не в
  HTML/JS хардкодом (тот же принцип, что RUKYA Pro `data/*.json`, только здесь
  content-модули курса).
- Следуй hard rules выше по ходу — особенно мобильный приоритет и единый источник
  токенов.

### 4. QA — проверка end-to-end (ОБЯЗАТЕЛЬНО, не пропускается)
- Для UI/сайта — **webapp-testing** (Playwright, реальный вьюпорт мобильный+десктоп)
  — не «код выглядит правильно», а реальный рендер.
- Для контента — независимая сверка по критерию §7/§9: перечитай написанное как
  если бы делал триаж заново (или раздай это отдельному `general-purpose`/
  `Explore`-агенту для непредвзятой сверки) — проверь, что нет пальпаторной
  диагностики, нет инструкций игнорировать страдание пациента, каждая цитата
  промаркирована по источнику.
- Красный результат → назад в стадию 3 (или в контентную правку). Не объявляй
  готовым с красным.
- Найденное, но не относящееся к текущей задаче — назови явно и спроси, чинить ли
  сейчас или отдельным маршрутом F.

### 5. Review
- **/code-review** на диф — баги корректности.
- **/security-review**, если тронута авторизация/оплата/интеграция с внешним API.
- Опционально **/simplify** после того, как корректность подтверждена.

### 6. Run + запись решения
- **/run** — поднять локальный предпросмотр сайта (или открыть страницу через
  `file://`/dev-сервер, в зависимости от того, что уже настроено) и подтвердить,
  что фича реально работает.
- Нетривиальное архитектурное решение (выбор бэкенда, структура модуля и т.п.) —
  запиши прямо в соответствующий раздел `project.md` (§16 открытые решения,
  §9а/б/в для контентных решений) — это и есть журнал решений этого проекта,
  отдельного adr-скилла здесь нет.
- Нетривиальный урок, который стоит помнить и в будущих сессиях (не только в этом
  проекте) — сохрани через собственную (глобальную) память по инструкции
  «auto memory» в начале системного промпта (feedback/project-типы), а не в
  `.claude/skills/`.

## Output discipline

- Формат итога — см. «Финальный итог» выше. Это единственный развёрнутый вывод.
- Пропущенную стадию назови строкой в итоге, коротко и с причиной.
- Провалы — честно, с `file:line` или названием раздела project.md. Никогда не
  заявляй «проверено», если стадия красная, и никогда не заявляй «сертифицировано»,
  если контент — черновик без подтверждения шейха.

## Scaling the ceremony

- **Мелкая правка** (текст на странице, одна кнопка, один стиль): стадии 1 (инлайн-
  план), 3, 4 (webapp-testing/сверка), 5 (code-review). Design check и запись
  решения — пропустить, если нет архитектурного выбора.
- **Средняя фича** (новый экран, новый флоу): все стадии.
- **Крупная/структурная** (новый модуль курса целиком, новая система оплаты/
  регистрации, смена бэкенда): все стадии, маршрут C для контентной части + маршрут
  E для портальной части параллельно, если они разделимы. Для ещё большего фан-аута
  (десятки под-задач, независимая перепроверка контента несколькими «голосами») —
  предложи **Workflow**, но только если пользователь явно согласен на
  multi-agent-оркестрацию.
